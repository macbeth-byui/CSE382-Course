CSE 382 Lesson 4 - Fold and Unfold Functor Design Patterns

The fold and unfold are part of the common functor design patterns. A fold will convert from a list to a single value whereas an unfold will convert from a single value and a list.

You can find the template for the problem sets in this lesson here: prove04.erl

Part 1 - Fold

Just like the map and filter patterns, we will use a lambda function to define what we want to do with each item of the list. Unlike the map and filter, we will not apply the lambda function to determine what to put in the resulting list. Instead (in the case of fold) we will use the lambda function to determine how each item in our list contributes to the one single value result. The lambda function is used to combine all the values in the list.

If I have a list of numbers [1 2 3 4 5] that I wanted to add, then a lambda function would take each number (ValueValueValue) and add it to an accumlator (AccAccAcc):

spec  λ::real  real→real.spec ~ ~ \lambda :: real ~ ~ real \rightarrow real.spec  λ::real  real→real.

def  λ::Value  Acc→Acc+Value.de\mathit{f} ~ ~ \lambda :: Value ~ ~ Acc \rightarrow Acc + Value.def  λ::Value  Acc→Acc+Value.

If I wanted to add the squares of the numbers in the list, then I would want the lambda function to be

spec  λ::real  real→real.spec ~ ~ \lambda :: real ~ ~ real \rightarrow real.spec  λ::real  real→real.

def  λ::Value  Acc→Acc+(Value∗Value).de\mathit{f} ~ ~ \lambda :: Value ~ ~ Acc \rightarrow Acc + (Value * Value).def  λ::Value  Acc→Acc+(Value∗Value).

The result of the lambda functions will be passed in as the accumulator value when we goto the next item in the list. This implies that we will need to define what the initial accumlator value should be.

Notice that the fold function is expecting that the lambda always has 2 input parameters where the second parameter is the accumulator. The single output of the λ\lambdaλ function is the updated accumulator.

Here is the formal definition of the fold:

spec  λ::a1  a2→a2.spec ~ ~ \lambda :: a_1 ~ ~ a_2 \rightarrow a_2.spec  λ::a1​  a2​→a2​.

spec  fold::λ  a2  [a1]→a2.spec ~ ~ \mathit{fold} :: \lambda ~ ~ a_2 ~ ~ [a_1]\rightarrow a_2.spec  fold::λ  a2​  [a1​]→a2​.

def  fold::λ  Acc  []→Acc;de\mathit{f} ~ ~ \mathit{fold} :: \lambda ~ ~ Acc ~ ~ []\rightarrow Acc;def  fold::λ  Acc  []→Acc;

def  fold::λ  Acc  [First∣Rest]→(fold  λ  (λ  First  Acc)  Rest).de\mathit{f} ~ ~ \mathit{fold} :: \lambda ~ ~ Acc ~ ~ [First|Rest] \rightarrow (\mathit{fold} ~ ~ \lambda ~ ~ (\lambda ~ ~ First ~ ~ Acc) ~ ~ Rest).def  fold::λ  Acc  [First∣Rest]→(fold  λ  (λ  First  Acc)  Rest).

Notice the recursive nature of the fold as it applies the lambda function to the each element one at a time starting with the first element (FirstFirstFirst). The result of calling the lambda function (λ  First  Acc\lambda ~ ~ First ~ ~ Accλ  First  Acc) becomes the new accumlator value when fold is called recursively on the remainder of the list (RestRestRest).

Consider the code implementation below.

fold(_Lambda, Acc, []) -> Acc;
fold(Lambda, Acc, [First|Rest]) -> fold(Lambda, Lambda(First, Acc), Rest).

The initial Acc passed to the fold function represents the initial value of the accumulator. If we were summing up numbers in a list, we would expect the initial accumlator to be 0.

Note that Erlang provides this function as a built-in function called lists:foldl (meaning fold left).

Part 2 - Fold Right

When you look at the definition of fold notice that the list is processed from left to right. We can define a function that goes from right to left called foldr. This function will require us to traverse to the end of the list before we can actually call the lambda function.

spec  λ::a1  a2→a2.spec ~ ~ \lambda :: a_1 ~ ~ a_2 \rightarrow a_2.spec  λ::a1​  a2​→a2​.

spec  foldr::λ  a2  [a1]→a2.spec ~ ~ \mathit{foldr} :: \lambda ~ ~ a_2 ~ ~ [a_1]\rightarrow a_2.spec  foldr::λ  a2​  [a1​]→a2​.

def  foldr::λ  Acc  []→Acc;de\mathit{f} ~ ~ \mathit{foldr} :: \lambda ~ ~ Acc ~ ~ [] \rightarrow Acc;def  foldr::λ  Acc  []→Acc;

def  foldr::λ  Acc  [First∣Rest]→(λ  First  (foldr  λ  Acc  Rest)).de\mathit{f} ~ ~ \mathit{foldr} :: \lambda ~ ~ Acc ~ ~ [First|Rest]\rightarrow (\lambda ~ ~ First ~ ~ (\mathit{foldr} ~ ~ \lambda ~ ~ Acc ~ ~ Rest)).def  foldr::λ  Acc  [First∣Rest]→(λ  First  (foldr  λ  Acc  Rest)).

The code implementation of our function in Erlang is left for an exercise below. Erlang provides a built-in function called lists:foldr to perform this task.

Part 3 - Unfold

The fold design pattern is used when you want to consolidate from one larger thing to a smaller thing such as a list to a single value. If we want to go backwards, this is called an unfold. Note that folding and unfolding are not inversely related. For example, if I had a list of numbers [2 5 3 1] and I folded them up using a simple sum function, I would get 11. However, if started with 11 and worked backwards, I could get a possible solution such as [7 1 3 0] which is different from our original list.

When we unfold, we are relying on some intial conditions and lambda function that is used to generate the next value for our list based on those initial conditions.

Suppose we wanted to generate a list of CountCountCount numbers with a value of ValueValueValue. We want to unfold our initial conditions to obtain a list of numbers. For example, if CountCountCount is 4 and ValueValueValue is 6, then the gensame function should result in [6 6 6 6]. Here is a definition for our gensame function:

spec  gensame::integer  real→[integer].spec ~ ~ gensame :: integer ~ ~ real \rightarrow [integer].spec  gensame::integer  real→[integer].

def  gensame::0  Value→[];de\mathit{f} ~ ~ gensame :: 0 ~ ~ Value \rightarrow [];def  gensame::0  Value→[];

def  gensame::Count  Value→[Value∣(gensame  (Count−1)  Value)].de\mathit{f} ~ ~ gensame :: Count ~ ~ Value \rightarrow [Value | (gensame ~ ~ (Count-1) ~ ~ Value)].def  gensame::Count  Value→[Value∣(gensame  (Count−1)  Value)].

Here is the Erlang and Clojure code for gensame:

gensame(0, _Value) -> [];
gensame(Count, Value) -> [Value | gensame(Count-1, Value)].

We could modify this function to create an increasing sequence of numbers by including an initial value (InitInitInit) and step size (StepStepStep). To accomplish this, we need to define a lambda function to calculate the next value. We will use CurrCurrCurr to represent the current value that started with InitInitInit and was increased by StepStepStep.

For example, if we wanted a list of 5 even numbers starting at 4, then CountCountCount would be 5, InitInitInit would be 4, and our lambda would be:

spec  λ::real→real.spec ~ ~ \lambda :: real \rightarrow real.spec  λ::real→real.

def  λ::Curr→Curr+2.de\mathit{f} ~ ~ \lambda :: Curr \rightarrow Curr + 2.def  λ::Curr→Curr+2.

The output of our genincr function would be [4 6 8 10 12].

spec  genincr::integer  real  λ→[real].spec ~ ~ genincr :: integer ~ ~ real ~ ~ \lambda \rightarrow [real].spec  genincr::integer  real  λ→[real].

def  genincr::0  Curr  λ→[];de\mathit{f} ~ ~ genincr :: 0 ~ ~ Curr ~ ~ \lambda \rightarrow [];def  genincr::0  Curr  λ→[];

def  genincr::Count  Curr  λ→[Curr∣(genincr  (Count−1)  (λ  Curr)  λ)].de\mathit{f} ~ ~ genincr :: Count ~ ~ Curr ~ ~ \lambda \rightarrow [Curr | (genincr ~ ~ (Count-1) ~ ~ (\lambda ~ ~ Curr) ~ ~ \lambda)].def  genincr::Count  Curr  λ→[Curr∣(genincr  (Count−1)  (λ  Curr)  λ)].

Looking at these two examples, we can generalize our function. Let’s create a generic unfold function that works for more than just numbers. This function does not exist in the Erlang library primarily because its not generic enough. We can create an unfold type function that does not rely on a Count variable or uses a lambda function that has a different number of parameters. The specification and definition is the same as genincr but will have a more generalized lambda specification with no definition.

spec  λ::a→a.spec ~ ~ \lambda :: a \rightarrow a.spec  λ::a→a.

The Erlang code for unfold is given below:

unfold(0, _Curr, _Lambda) -> [];
unfold(Count, Curr, Lambda) -> [Curr|unfold(Count-1, Lambda(Curr), Lambda)].
